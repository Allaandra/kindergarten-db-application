# Kindergarten App

Вебінтерфейс для роботи з базою даних дитячого садочка.  
Проєкт створено в рамках курсової роботи, для навчальних цілей.

- **ПІБ студента:** *Матвєєнко Олександра*  
- **Група:** *ІПЗ-3.03*  

## Опис проєкту

Система надає простий інтерфейс для роботи з даними дитячого садочка, які зберігаються в реляційній базі даних (PostgreSQL).  
Код інтерфейсу та частина бекенду писалися поспіхом і частково за допомогою ШІ, тому структура далека від продакшн‑рівня і проєкт спеціально залишений “як є” для демонстрації та подальшого рефакторингу.

Основні ролі користувачів:

- **Адміністратор** – керує групами, співробітниками, дітьми, родичами, розкладом, меню, журналами.
- **Вихователь** – працює зі своїми групами та відвідуванням дітей.
- **Медсестра** – веде медичний журнал дітей.
- **Батьки** – переглядають дані про свою дитину (відвідування, медичні записи, меню, розклад).

## Функціональність

- Управління **групами** (назва, вік, вихователь, заповненість).
- Управління **співробітниками** (посада, контакти, облікові дані).
- Управління **дітьми** та їх **родичами/опікунами**.
- **Розклад занять** по групах.
- **Меню харчування** та **довідник страв**.
- **Журнал відвідування** дітей.
- **Медичний журнал** (події, зауваження, огляди).
- Кабінети для різних ролей з відповідними правами доступу.

## Технології

### Backend

- **Node.js**, **Express**
- **PostgreSQL**
- Прямі SQL‑запити без ORM
- Розділення на шари:
  - маршрути (routes)
  - контролери (controllers)
  - робота з БД (dao / db context)
  - DTO / меппінг результатів запитів

### Frontend

- **JavaScript** / **TypeScript** (залежно від конфігурації)
- Фреймворк **React** (або інший SPA‑фреймворк, використаний у проєкті)
- Прості CSS‑стилі / інлайн‑стилі без складної дизайн‑системи
- Розподіл на сторінки за ролями: адмін, вихователь, медсестра, батьки

## Структура репозиторію (загальна)

- `backend/` – серверна частина:
  - `controllers/` – логіка обробки запитів
  - `routes/` – налаштування маршрутів API
  - `dao/` – підключення до БД, виконання SQL‑запитів
  - `dto/` – перетворення даних між БД та API
- `frontend/` – клієнтська частина:
  - сторінки для різних ролей (admin / educator / nurse / parent)
  - компоненти інтерфейсу
  - стилі

(Назви папок можуть трохи відрізнятися залежно від поточної структури в цьому репозиторії.)

## Запуск проєкту локально

### 1. Налаштування бази даних

1. Створити базу даних у PostgreSQL.
2. Виконати SQL‑скрипти зі структурою таблиць (якщо є в репозиторії).
3. Створити користувача/роль для підключення застосунку.

### 2. Backend

```bash
cd backend
npm install
npm run start   # або: node server.js / npm run dev
```

Перед запуском необхідно створити файл `.env` з параметрами підключення до БД, наприклад:

```env
DB_HOST=localhost
DB_PORT=5432
DB_NAME=kindergarten
DB_USER=your_user
DB_PASSWORD=your_password
PORT=3000
```

### 3. Frontend

```bash
cd frontend
npm install
npm run dev
```

Після цього відкрити в браузері адресу, яку виведе dev‑сервер (зазвичай `http://localhost:5173` або подібну).

## Статус коду та обмеження

- Навчальний проєкт у рамках курсової роботи.
- Код місцями дублюється, майже немає валідації та обробки помилок.
- Частина логіки “захардкоджена” у компонентах.
- Безпека (аутентифікація, авторизація, шифрування паролів) реалізована спрощено або частково.

Проєкт можна використовувати як базу для:

- захисту курсової роботи;
- тренування з проєктування БД;
- відпрацювання повного циклу: БД → backend API → frontend;
- подальшого рефакторингу (структура, патерни, тестування, безпека).
